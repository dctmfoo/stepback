## 1. Problem

Milestone 3 shipped browsing, but composing is still primitive: "New Routine" creates an empty routine and pushes its detail; there is no way to add, edit, reorder, or time steps, and "New Routine" from the gallery creates a one-step routine directly (Milestone 3 D12 explicitly deferred the real flow to this milestone). PLAN.md Milestone 4 requires the make-or-break screen (PRD §0.3, §5.4): name field, step rows with mint rest rows, inline step editor in 5 s increments, drag reorder, a multi-select gallery picker with smart defaults, the floating glass Add-Workouts/total bar, and a live total from the shared compiler. The gate is PRD acceptance #2 (build the §5.4 sample routine naturally) plus ui-spec §builder conformance.

## 2. Goals

- G1. **Builder screen** per ui-spec §builder: Cancel / title / Save navigation, Name card, ordered step list with quiet Recover-Mint rest rows *between* steps, drag-handle reorder, tap-to-expand inline step editor.
- G2. **Gallery picker** per ui-spec §builder second frame: category chips + search + multi-select rows with azure checks, a visible selection tray, one commit button ("Add N Workouts"); sheet on compact, side-by-side pane in regular width (DESIGN.md §5).
- G3. **Smart defaults with carry-forward** (PRD §5.4): new steps land with 30 s work · 1 set · no set-rest · 15 s rest-after, and the previous step's values become the defaults for the next add within a builder session.
- G4. **Floating glass bar** — Add Workouts + live computed total, one of the app's two permitted custom glass elements (DESIGN.md §6), always visible while composing.
- G5. **Live total from the shared compiler**: the bar's figure is `TimelineCompiler` output over the draft, integer seconds, tabular numerals — the same figure routine detail and cards show (PRD §6.2).
- G6. **All three entry points wired:** Routines home "New Routine", routine detail "Edit" (pre-populated), gallery Add-to-Routine "New Routine" (workout pre-loaded as step 1).
- G7. **Localization + accessibility:** every new string in `Localizable.xcstrings` (§6 table), accessibility identifiers per §6, VoiceOver grouping for step rows and adjustable editor controls, Dynamic Type through AX sizes.
- G8. **Gate:** UI test builds the PRD §5.4 sample routine end-to-end; reorder and edit covered; totals match compiler output; ui-spec §builder conformance check (PLAN.md Milestone 4 gate, PRD acceptance #2).

## 3. Non-goals

- **No player** — Milestone 5. Save returns to where the user came from; Play still routes through the Milestone 3 `PlayerLauncher` stub.
- **No new workout-creation surface** — the picker links to the existing custom-workout editor sheet (Milestone 3 D13) via the gallery's established "Add your own" affordance if reachable without clutter; otherwise users add custom workouts in the Gallery tab first. The picker never grows its own workout form.
- **No per-set distinct timing, supersets, circuits, or grouping** — PRD §2 non-goals (no programs); a step is one workout with uniform set timing, exactly the PRD §4.3 shape.
- **No free-text duration entry** — steppers only (PRD §5.4, DESIGN.md §4); no keyboard for seconds, ever.
- **No routine-level settings** (get-ready override, per-routine audio) — Settings owns get-ready (PRD §5.6, Milestone 5); the builder edits name + steps, nothing else.
- Checked against PRD §2: multi-select, carry-forward defaults, and the live total are all §5.4 requirements, not scope growth; nothing here adds programs, coaching, or dashboards.

## 4. Design decisions

- **D1. The builder is a full-height system sheet over its presenter, with Cancel / inline title / Save.** Matches the ui-spec mockup's nav row ("Cancel · Edit Routine · Save") and DESIGN.md §6 ("Sheets are system: detents, grabber, content peeking beneath") — on iPhone/iPad a full-height sheet, on Mac the standard sheet presentation. Titles: "New Routine" / "Edit Routine" per entry point. Rejected: a pushed navigation destination (Cancel/Save is modal grammar; a push would put Save behind a back button and allow half-committed navigation away) and `fullScreenCover` (hides context for no benefit; the sheet is the established modal idiom from Milestone 3's editor sheets).
- **D2. All editing happens on an in-memory draft; Save commits once via the save-or-rollback path.** The builder never mutates SwiftData models while composing. It builds a draft (name + ordered step values, including `workoutID` and `workoutNameSnapshot` captured at add time) and on Save maps the draft onto the `Routine`/`RoutineStep` models in one transaction through the existing `ModelContext` save-or-rollback helper (Milestone 3 hardening), firing the `.success` haptic (DESIGN.md §7 `routine_saved`). Cancel discards the draft. Rationale: live-editing models would stream half-built routines to CloudKit mirroring and make Cancel a manual undo problem; the draft makes Cancel trivially safe and Save atomic. Editing an existing routine reuses its `RoutineStep` rows where possible on commit (update in place by step identity, insert/delete the difference) so CloudKit sync churn stays proportional to the edit.
- **D3. Entry points.** (a) Routines home "New Routine" presents the builder with an empty draft and the suggested default name (existing `routine.default-name` + `RoutineLibrary.nextAvailableRoutineNumber` — PRD §5.4 "suggest a default name"); it no longer creates-then-pushes an empty routine (that Milestone 3 D9 behavior existed only because the builder didn't). On Save, the new routine's detail is pushed. (b) Routine detail gains **Edit**, which presents the builder pre-populated from the routine (ui-spec §routine-detail promised Edit "opens the builder pre-populated"; Milestone 3 D8 deferred it here). (c) `AddToRoutineSheet`'s "New Routine" presents the builder with the originating workout as step 1 under smart defaults (PRD §5.3/§5.4), replacing Milestone 3 D12's direct one-step creation. The "add to *existing* routine" path in that sheet keeps its Milestone 3 behavior (append with smart defaults, no builder detour) — it is a one-tap checkout, not a composing session.
- **D4. Step list anatomy reuses the Milestone 3 row family.** Each step: drag handle · `WorkoutVisual` small tile · workout name · timing summary footnote (existing `routine.set-summary` grammar: "30 s × 3 · 10 s between sets", "~20 reps" appended when guidance is set — same formatters as routine detail, so the two screens can never drift) · disclosure chevron. `RoutineStepRow` and `RoutineRestRow` are the visual source; the builder composes them with handle + disclosure rather than cloning them (DESIGN.md §4 "Builder step row" names exactly this anatomy). Rest-after renders *between* rows as the quiet Recover-Mint row ("Rest · 15 s", existing `routine.rest` key) — Recover Mint is rest-only (DESIGN.md functional-hue contract), tappable to expand its owning step's editor.
- **D5. The final step's rest row is not rendered (conformance note, not an amendment).** The ui-spec mockup and the PRD §5.4 sample both end on a step, and the compiler drops trailing rest ("a routine never ends on a rest", PRD §6.2, pinned by Milestone 1 tests). The rest-after *value* is preserved on the step, so reordering a step away from last position restores its rest row — rest-after travels with its step (PRD §5.4 "rest-after values travel with their step"). The expanded editor always shows the Rest-after control, even on the last step, with a quiet footnote that it plays only when another step follows (§6 key) — honest, and it explains why the list shows nothing after the last row.
- **D6. Expanded inline editor: five labeled stepper rows, expanding in place.** Tapping a step (or its rest row) expands controls directly beneath it, exactly the ui-spec "stepedit" block: **Work** (5 s steps, 5 s–10 min), **Sets** (1 step, 1–20), **Rest between sets** (5 s steps, 0 s "Off"–5 min), **Rep guidance** (Off, then ~5–~100 in steps of 5 — stored as the optional `repGuidance` Int, Off = nil), **Rest after** (5 s steps, 0 s–5 min). One step expanded at a time; expanding another collapses the first (the mockup's "Bridge expanded" state; keeps the one-hero-total screen calm). While a step is expanded, its rest row is absorbed into the editor ("it renders there instead of as a row" — ui-spec caption). Controls are system `Stepper`s with the value displayed beside the label in tabular numerals. Grounding: `Stepper` is available iOS 13+/macOS 10.15+ (docset `/documentation/swiftui/stepper`, Platforms line read 2026-07-10); `WheelPickerStyle` is **not available on macOS** (docset `/documentation/swiftui/wheelpickerstyle` — iOS/iPadOS/Catalyst/visionOS/watchOS only), so steppers are the single cross-platform control — PRD §5.4 allows "steppers/wheels" and DESIGN.md §4 closes with "Segmented/steppers/pickers: system controls, system styles, always". Rejected: wheels on iOS + steppers on Mac (two behaviors to test and describe for zero user benefit); free-text entry (forbidden, PRD §5.7).
- **D7. Ranges clamp, never error.** Work minimum 5 s (a 0 s work segment is meaningless), sets minimum 1, rests minimum 0 s. Upper bounds (10 min work, 5 min rests, 20 sets, ~100 reps) are generous stepper rails, not validation — there is no way to build an invalid routine (PRD §5.4). Steppers disable their −/+ at the rails, which is the system-native signal.
- **D8. Reorder: standard list reorder with always-visible drag handles; no Edit-mode toggle.** The mockup shows grip glyphs on every row in the resting state — reorder is a first-class composing action, not a mode. The implementation uses the standard SwiftUI list move mechanism (`onMove(perform:)`, iOS 13+/macOS 10.15+ — docset `/documentation/swiftui/dynamicviewcontent/onmove(perform:)`, Platforms line read 2026-07-10) with the row kept permanently movable; `EditButton` is **not** part of this design and is unavailable on macOS anyway (docset `/documentation/swiftui/editbutton` — iOS/iPadOS/Catalyst/visionOS only). Moving a step carries its rest-after with it (D5); an expanded editor collapses when a drag begins. Step delete lives on the row (swipe on iOS, context menu everywhere — system idiom; swipe-to-delete is fine here because a *step* is cheap to re-add, unlike the routine-card delete DESIGN.md §5 restricts).
- **D9. Floating glass bar via a bottom safe-area inset.** One capsule bar floating above the list: **Add Workouts** (leading, with plus icon, PulseAzure text) and **Total** with the live figure (trailing, tabular semibold). This is one of the exactly two custom glass elements (DESIGN.md §6 glass budget; ui-spec caption names this bar as one of them). It is inset into the bottom safe area so the list scrolls beneath it and the last rows remain reachable (`safeAreaInset` verified iOS 15+/macOS 12+ — docset `/documentation/swiftui/view/safeareainset(edge:alignment:spacing:content:)`, Platforms line read 2026-07-10). The bar is always visible while composing (PRD §5.4 "always-visible Add Workouts affordance" + "live total always visible").
- **D10. The live total is compiled, not summed.** On every draft change the builder maps the draft to the core `RoutineSnapshot` and reads `TimelineCompiler.compile(_:getReadySeconds: 0).totalDurationSeconds` — byte-identical to routine detail's hero figure (Milestone 3 wiring) and the card totals. No UI-side arithmetic, ever (PRD §6.2 "the compiled timeline is the single source of truth for every displayed total"; ui-spec §routine-detail note "if they ever disagree, that's a bug"). Total renders via the existing `DisplayFormatters.duration`, changes with the numeric content transition (Milestone 3 D16 baseline), and the whole-bar VoiceOver value uses `DisplayFormatters.spokenDuration`.
- **D11. Gallery picker: sheet on compact, side-by-side pane in regular width.** Compact: a system sheet (grabber, medium + large detents per the established Milestone 3 sheet pattern) containing search, category chips, and the multi-select list. Regular width (iPad landscape, Mac window): the builder pairs the step list with the picker side-by-side instead of a covering sheet (DESIGN.md §5 "the builder pairs the step list with the picker in a side-by-side arrangement when width allows"; ui-spec §builder note 7) — the picker pane opens on "Add Workouts" and closes on commit or dismiss. Size classes only, no device-type branches (PRD §3). Content in both containers is the same component: category chips (All + eight categories, selected chip PulseAzureSoft — ui-spec caption), `.searchable`-style search field scoped to the picker, rows of `WorkoutVisual` tile + name with an azure check on selection, custom workouts inline with the "Yours" caption (existing gallery grammar).
- **D12. Multi-select with a visible tray and one honest commit button.** Selection order is preserved — steps land in the order the user picked them. The tray shows the selected workouts' tiles (scrolling horizontally past ~6) with a "Selected" label; the commit button reads "Add 1 Workout" / "Add N Workouts" (plural key, §6) — the button says exactly what it does, never "Continue"/"Done" (DESIGN.md §8; ui-spec caption). Tapping a selected row deselects it; tapping a tray tile also deselects (the tray is the receipt, and receipts are editable). Commit appends all selections as steps (smart defaults per D13), closes the picker, and scrolls the list to the first new step. Selection fires the `.selection` haptic (DESIGN.md §7 pickers rule). An empty selection renders the commit button disabled rather than hidden.
- **D13. Smart defaults and carry-forward.** First add in a builder session: 30 s work · 1 set · 0 s set-rest · 15 s rest-after · guidance Off (PRD §5.4). After any step exists, the defaults for newly added steps are **the last step's current values** (work, sets, set-rest, rest-after — not rep guidance, which is per-exercise by nature and always defaults to Off) because "routines are rhythmic" (PRD §5.4; ui-spec note 1). "Last step" means the final step in the list at commit time; a multi-select batch applies the same defaults to every step in the batch (the batch is one gesture — making step 3 inherit from step 2 of the same batch would surprise). `workoutNameSnapshot` is captured at add time (PRD §4.3 contract).
- **D14. Save validates one thing only.** Save is enabled iff the draft has ≥ 1 step and a non-empty trimmed name; since the name field is pre-filled with the suggested default (D3) and steppers cannot produce invalid values (D7), the only reachable disabled state is the zero-step draft (PRD §5.4). The empty step list shows a small designed hint in the list area ("Add workouts to build your routine" — §6 key; `ContentUnavailableView` shape per DESIGN.md §4 empty-state rule) pointing at the floating bar, not a bare void.
- **D15. Cancel confirms only when the draft is dirty.** Cancel (and the sheet's interactive dismiss) discards silently when the draft equals its initial state; when dirty, a confirmation dialog offers Discard Changes (destructive role — the system-sanctioned destructive use, consistent with DESIGN.md's no-red rule exception) / Keep Editing. Interactive dismiss is blocked while dirty so a stray swipe cannot destroy ten minutes of composing.
- **D16. `@Observable` main-actor draft model; formatters and category styling reused.** The draft lives in an `@Observable` builder model (house standard, PRD §3), owning the step array, expansion state, picker selection, and the compiled total. All display strings via the existing `DisplayFormatters`/`L10n` surface; category hues/symbols via the existing `CategoryStyle`; workout identity resolution (catalog + custom + missing-id fallback) via the existing `WorkoutItem`/`WorkoutLibrary` path — the builder introduces zero new visual identities (PRD §4.6: every workout visual through `WorkoutVisual`).

## 5. Edge cases

Composing:
- Zero-step draft: Save disabled, designed hint shown (D14); Cancel exits silently if nothing else changed.
- Deleting the only remaining step returns to the zero-step state (Save disables again); deleting any step removes its rest row with it.
- Reordering the last step upward re-reveals its preserved rest row; reordering another step into last position hides that step's rest row (D5). Rest values are never zeroed by reorder.
- Two steps referencing the same workout are legal and independent (the §5.4 sample pattern generalizes; steps are value rows, not references to a set).
- A workout deleted from the gallery (custom) while the builder is open: existing steps keep rendering via `workoutNameSnapshot` and the `WorkoutVisual` missing-id fallback (PRD §4.1/§4.3); the picker no longer offers it. No crash, no blank row.
- Rep guidance renders in the summary only when set ("30 s · ~20 reps"); Off shows nothing — no "0 reps".
- Set-rest 0 s renders no "between sets" clause in the summary (existing Milestone 3 grammar).

Picker:
- Search with no matches: system search-empty state within the picker (Milestone 3 pattern).
- All workouts already in the routine: no restriction — the library is not consumed by use.
- Regular-width pane open while the window narrows to compact (iPad multitasking): the pane's selection state transfers to the sheet presentation, or (acceptable) the picker dismisses preserving nothing if selection was empty; whichever is implemented, no selection is silently committed. Size-class transition must not crash or double-present.
- Commit of N selections lands N steps atomically in the draft (a single undo-visible mutation — relevant only to the dirty flag, D15).

Editing an existing routine:
- Save maps by step identity: edited steps update, removed steps delete, added steps insert with fresh identity (D2); the routine's `updatedAt` bumps once.
- Cancel after edits leaves the persisted routine byte-identical (draft isolation, D2) — detail totals unchanged.
- A sync-delivered change to the same routine while the builder is open: last-writer-wins at Save (CloudKit mirroring semantics, PRD §3); acceptable for a single-owner private-DB app, noted so nobody builds merge UI.

Entry points:
- Gallery "New Routine" with a custom workout: step 1 snapshots its current name; later renames of the custom workout do not rewrite the snapshot (PRD §4.3, pinned since Milestone 3).
- Save from the home entry pushes the new routine's detail; Save from detail-Edit returns to the (refreshed) detail; Save from the gallery entry dismisses to the gallery with the routine created (its home card appears on next visit) — no double navigation.

## 6. Accessibility & localization

**New string-catalog keys** (English values; existing keys `common.save`, `common.cancel`, `custom.name`, `routine.rest`, `routine.set-summary`, `routine.reps`, `routine.default-name` are reused, not duplicated):

| Key | English value | Notes |
|---|---|---|
| `builder.title.new` | New Routine | sheet title, home + gallery entries |
| `builder.title.edit` | Edit Routine | sheet title, detail entry |
| `builder.name.label` | Name | name card caption (uppercase styling is visual, not in the string) |
| `builder.add-workouts` | Add Workouts | floating bar + picker pane title |
| `builder.total` | Total | floating bar label |
| `builder.empty.title` | No workouts yet | zero-step hint title |
| `builder.empty.message` | Add workouts to build your routine. | zero-step hint body |
| `builder.step.work` | Work | editor row label |
| `builder.step.sets` | Sets | editor row label |
| `builder.step.set-rest` | Rest between sets | editor row label |
| `builder.step.rep-guidance` | Rep guidance | editor row label |
| `builder.step.rest-after` | Rest after | editor row label |
| `builder.step.off` | Off | rep-guidance / set-rest zero position |
| `builder.step.rest-after.footnote` | Plays only when another step follows. | last-step rest explainer (D5) |
| `builder.step.delete` | Remove Step | row delete action |
| `builder.discard.title` | Discard changes? | dirty-cancel dialog |
| `builder.discard.confirm` | Discard Changes | destructive role |
| `builder.discard.keep` | Keep Editing | |
| `builder.picker.selected` | Selected | tray label |
| `builder.picker.add-count` | Add %lld Workout / Add %lld Workouts | plural variants required |
| `builder.step.accessibility` | %1$@, %2$@ | VoiceOver combined step label: name, spoken summary |

Durations inside steppers, summaries, and the total come from `DisplayFormatters` (Foundation) — never the catalog (PRD §3.1). Rep values reuse the existing `routine.reps` ("~%lld reps") grammar.

**Accessibility identifiers:** `builder.name`, `builder.name.label`, `builder.empty`, `builder.save`, `builder.cancel`, `builder.addWorkouts`, `builder.total`, `builder.step.<index>` (row), `builder.step.<index>.rest` (rest row), editor controls `builder.step.<index>.work|sets|setRest|repGuidance|restAfter`, `builder.picker.search`, `builder.picker.chip.<categoryID>`, `builder.picker.row.<workoutID>`, `builder.picker.tray`, `builder.picker.add`, `builder.discard.confirm`, plus `detail.edit` on the routine detail toolbar.

**VoiceOver:** each collapsed step row is one grouped element — label from `builder.step.accessibility` (name + spoken timing summary via the existing spoken-duration path), with expand, delete, and move exposed as custom/row actions; the rest row is grouped with the spoken "Rest, 15 seconds" grammar already shipped in Milestone 3. Stepper rows expose the system adjustable trait with the current value spoken ("Work, 30 seconds"). The floating bar reads as two elements: the Add Workouts button and a "Total, 5 minutes 30 seconds" static value that updates. Picker rows announce selection state; the commit button's label carries its live count.

**Dynamic Type:** rows follow Dynamic Type with no exceptions (the stage-numerals exception does not apply here). At AX sizes the editor's label + stepper stack vertically per system behavior; the floating bar's two elements wrap to two lines rather than truncating the total (the total is never clamped — it is the screen's one live number). Verified at AX XXXL in the gate.

## 7. Test impact

- **StepBackCore:** no changes — the compiler already covers the §5.4 sample and trailing-rest behavior (Milestone 1). The builder must not add math anywhere testable outside core.
- **App unit tests (`StepBackTests`):** draft round-trip (routine → draft → save → identical snapshot); edit-in-place identity mapping (D2: edited/removed/added steps); smart-default carry-forward incl. batch semantics and rep-guidance exclusion (D13); draft total equals `TimelineCompiler` output for the §5.4 sample; dirty-flag behavior backing D15; snapshot capture at add time.
- **UI tests (`StepBackUITests`):** the gate test — build the full PRD §5.4 sample routine through the real UI (add five workouts via multi-select where possible, set every timing via steppers, verify the floating-bar total equals the known compiled figure, Save, verify detail hero shows the same figure); a reorder test (move a step, assert its rest row traveled and the total is unchanged); an edit test (open Edit, change one value, Save, assert detail updated; Cancel path asserts no change). iPad lane exercises the regular-width side-by-side picker.
- **Mac (`StepBackMacUITests`):** builder opens from New Routine, a step can be added and saved (smoke — reorder/steppers are shared code proven on iOS lanes).
- Existing Milestone 3 tests that pinned "New Routine creates an empty routine and pushes detail" and gallery D12 direct-creation behavior must be updated to the new builder-first flows — expected churn, listed so it isn't mistaken for regression.

## 8. Acceptance criteria

1. From Routines home, "New Routine" opens the builder with a suggested name; the PRD §5.4 sample routine can be composed exactly as written — five workouts in one picker pass, per-step timing via steppers in 5 s increments, rest rows reading between steps — and saved (PRD acceptance #2).
2. The floating bar's total is always visible, updates with every change, and equals the compiled-timeline figure shown on the saved routine's detail and card — one source of truth, integer seconds, tabular numerals.
3. Reordering via drag handles carries each step's rest-after with it; the final step never shows a trailing rest row; no reorder can change the compiled total.
4. Edit from routine detail opens the builder pre-populated; Save applies exactly the edits; Cancel (including attempted swipe-dismiss) leaves the persisted routine untouched, with a discard confirmation only when dirty.
5. Gallery → workout → Add to Routine → New Routine opens the builder with that workout as step 1 under smart defaults; add-to-existing keeps its one-tap Milestone 3 behavior.
6. Save is possible the moment one step exists and never blocked by any other condition; the zero-step state shows the designed hint and a disabled Save.
7. The picker presents as a sheet in compact and side-by-side in regular width, with multi-select, tray, live-count commit button, search, and category chips; selection order equals resulting step order.
8. Every new user-facing string resolves from `Localizable.xcstrings` (pseudolocalization render check passes); all §6 accessibility identifiers exist; VoiceOver walks the builder per §6 grouping; AX XXXL layout holds.
9. ui-spec §builder conformance verified against the running app on iPhone and iPad (design-spec preview side-by-side); glass budget still exactly two custom elements.
10. `make test-core`, `make test-app`, `make test-ipad`, `make test-mac` green; `make gen` idempotent; PLAN.md Milestone 4 gate closed citing the green commit.
