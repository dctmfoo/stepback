## 1. Problem

Milestones 0–6 built the entire product, and each gate verified its own slice — but nobody has ever walked the whole thing. PLAN.md Milestone 7 is the closing checkpoint: cross-device sync verification (PRD acceptance #10), Mac end-to-end (#12), VoiceOver end-to-end (#11), pseudolocalization + string-catalog audit (#13), the PRD §8 performance targets, and timer-drift measurement — then walk every PRD §9 acceptance criterion, record results, and tag `v1-candidate`.

Specifically outstanding today:

- **No end-to-end acceptance record exists.** Criteria #1–#9 were each demonstrated inside their milestone, but never re-verified together on one build; #10–#13 have never been verified at all.
- **PRD §8 numbers have never been measured.** Cold launch < 1 s, play-tap → pre-roll < 300 ms, smooth gallery scrolling with the full catalog, and ≤ 150 ms per-segment timer drift with no cumulative drift are all asserted by architecture (monotonic recompute, compiled timeline) but not by measurement.
- **Accessibility has been spot-checked per milestone** (VoiceOver groupings, AX-size layouts) but there is no automated audit lane and no single end-to-end VoiceOver walk of build → play → complete.
- **String-catalog discipline is enforced by review, not by a check.** Pseudolocalization renders were done manually in M3/M6; nothing prevents a hardcoded string from landing tomorrow.
- **CloudKit sync has never been exercised across two physical devices** (M2's gate noted "when feasible"; it wasn't).

## 2. Goals

- G1. **The acceptance walk**: every PRD §9 criterion (#1–#13) verified against one identified build and recorded with evidence in a committed acceptance record (D1).
- G2. **Automated accessibility audits** on every primary screen, in the existing iPhone/iPad/Mac UI-test lanes, with only the documented, evidence-backed exemption allowlist (D2 and the accessibility-unblock amendment).
- G3. **Measured performance**: cold launch, play-tap → pre-roll latency, and gallery scroll behavior measured — on physical devices for the acceptance verdicts, via a repeatable opt-in test lane (D3, D4).
- G4. **Measured timer integrity**: the player instrumented with signposts; per-segment and cumulative drift measured over the PRD §5.4 sample routine on the physical iPad; a real-clock drift test pinned in the automated suite (D5).
- G5. **String-catalog audit automated**: a source scan that fails the unit-test lane on user-facing literals outside the catalog access layer, plus the established catalog-compile and pseudolocalization checks (D6).
- G6. **Two-device sync verification** of routines, custom workouts, session history, starter-seeding dedupe, and the Settings sync line, on the owner's physical iPhone + iPad (D7).
- G7. **Mac end-to-end** automated walk (browse → build → play with keyboard controls) plus the owner-visible manual pass (D8), and the **VoiceOver end-to-end** manual walk (D9).
- G8. **Gate**: all 13 acceptance criteria PASS in the record; all test lanes green; repo tagged `v1-candidate` (D11).

## 3. Non-goals

- **No new features, screens, flows, or user-facing strings.** The milestone's only product-code change is invisible instrumentation (D4). Anything the walk reveals as a *design gap* — specified behavior that is wrong or missing — halts that criterion and goes back through the planning agent (D10); this spec does not license inventing behavior to make a criterion pass.
- **No speculative performance optimization.** Code changes for performance happen only against a measured miss of a PRD §8 target, as defects with before/after numbers — never "while we're in there".
- **No CI/CD, App Store, or TestFlight work.** `v1-candidate` is a git tag, not a submission; screenshots, marketing copy, and archive/notarization pipelines are out of scope (PRD §2 spirit: no accounts, no services — and PLAN.md ends at the tag).
- **No new stats, dashboards, or verification UI.** The acceptance record is a markdown document, not an in-app surface (PRD §2 non-goals; §7 "glanceable, never a dashboard").
- **No third-party testing/snapshot frameworks.** XCTest, XCUIAutomation, and the unified logging system cover everything below; adding a dependency for verification would itself be new risk.
- **No re-litigating closed gates.** M0–M6 evidence stands; the walk re-verifies *outcomes* on the final build, not the milestones' internal red/green history.
- Checked against PRD §2: nothing here adds programs, media content, HealthKit, social, or network surface; the milestone only proves what §9 already demands.

## 4. Design decisions

**The acceptance record**

- **D1. Acceptance evidence lives in one committed document: `docs/verification/v1-acceptance.md`.** One row per PRD §9 criterion: verdict (PASS / FAIL / BLOCKED), the evidence (test name, measurement with units, or session-journal reference for manual walks), the build (short commit hash) it was verified against, the device/simulator used, and the date. PRD §8 measurements (launch, latency, drift, scroll) get their own table in the same document with target vs. measured. Rationale: PLAN.md M7 says "walk every PRD §9 acceptance criterion and record results" — session journals are session-scoped and THIN by design (sessions/README.md), while the acceptance record is product-scoped and must survive as the single durable answer to "was v1 actually verified, on what, and when". Rejected: recording results only in the closing session journal (scattered across sessions, violates THIN); a PLAN.md results table (PLAN carries sequencing and gates, not evidence — CLAUDE.md); a per-criterion checklist inside this spec (specs are frozen at Ready; evidence accretes during implementation).
- **D2. Automated accessibility audits via `performAccessibilityAudit`, in the existing UI-test lanes, with a reviewed allowlist.** `XCUIApplication.performAccessibilityAudit(for:_:)` is available iOS 17+ / iPadOS 17+ / **macOS 14+** (docset `/documentation/xcuiautomation/xcuiapplication/performaccessibilityaudit(for:_:)`) — inside this project's 26.0 floor on all three platforms, so the iPhone, iPad, *and* Mac lanes all audit. Audited screens: Routines home, routine detail, gallery, workout detail, builder (with picker open), Settings, welcome, and the player stage in a work and a rest segment. Audit types: the full set. The original pre-approved exception remains **dynamic-type findings on `player.countdown`**, the design system's sole fixed-size exception. The accessibility-unblock amendment adds a narrow allowlist for demonstrated XCTest/system-owned findings: wholly offscreen pixel findings or pixels wholly contained by identified native chrome; exact native tab/sidebar controls; the native searchable field's AX-XXL crop report constrained by element type; exact disabled Save/Add contrast; picker labels outside their named viewport; exact semantic-child crops contained by identified controls/cards/section headers; the combined iCloud Form node; no-element detection reports; noninteractive native Mac containers identified by type; Touch Bar; and the proven native get-ready action report. Predictive clipping warnings and exact logged SwiftUI semantic-font Dynamic Type signatures are allowed only because the same suite audits actual visible text clipping at AX-XXL on every required surface. App-owned findings and arbitrary geometry overlap are never filtered. Any growth requires a spec amendment and matching test evidence. **“Always-on” describes what the closing lane contains, not how often agents run it:** follow CLAUDE.md's mandatory ladder and run the full Mac lane only once at closeout. Rejected: a separate audit-only target; auditing only iOS; treating audits as a substitute for the human VoiceOver walk.

**Performance & timing measurement**

- **D3. Performance tests are an opt-in suite inside `StepBackUITests`, driven by a new Makefile lane, with acceptance verdicts taken on physical devices.** Cold launch is measured with `XCTApplicationLaunchMetric` (docset `/documentation/xctest/xctapplicationlaunchmetric` — Platforms: iOS/iPadOS/Mac Catalyst/tvOS/visionOS/watchOS; **no native macOS**, so launch and signpost metrics run in the iPhone/iPad lanes only, and the Mac's §8 story is covered by D8's end-to-end walk feeling native, not by a number PRD never set for it). Play-tap → pre-roll is measured with `XCTOSSignpostMetric(subsystem:category:name:)` over the D4 signpost region (docset `/documentation/xctest/xctossignpostmetric` — same iOS-family platforms; measures the time between matching begin/end events). Gallery scroll with the full ≥ 80-workout catalog plus custom workouts is verified on device by eye and Instruments during the acceptance walk — a simulator FPS number would be theater. The suite is excluded from the default lanes (measurement iterations are slow and variance on loaded CI machines makes red/green meaningless) and enabled by an environment opt-in mirroring the established `StepBackUITesting`/`StepBackUIShowWelcome` seam pattern; the new `make` lane runs it against a chosen destination, including the paired physical devices (device identifiers already live in the Makefile). CLAUDE.md §Commands gains the lane (header note). Rejected: perf tests in the default lanes (slow, flaky-by-variance); a new xcodegen test target (project churn for what an env seam does); trusting simulator numbers for the < 1 s / < 300 ms verdicts (PRD §8 says *target devices*; simulator runs are recorded as indicative only).
- **D4. The player gains signpost instrumentation behind one small utility — no behavior change, negligible cost, permanent value.** `OSSignposter` (docset `/documentation/os/ossignposter` — iOS 15+/macOS 12+; interval begin/end plus point events) intervals, subsystem = the app's bundle identifier, category `player`: (a) **play-tap → pre-roll ready** — begins where the Play action fires, ends when the pre-roll segment is presented; this is the D3 latency region and the PRD §8 "< 300 ms" contract made observable; (b) **one interval per timeline segment** — begin/end at the production transition moments, carrying the segment's compiled offset so drift is inspectable in Instruments on any future build. Instrumentation lives in production code (it must measure the real player, and signposts are designed to ship), sits behind the same protocol-with-fake discipline as every other service (the deterministic `PlayerSessionModel` tests keep using a no-op), and touches no layout, string, or token — which is why this milestone stays feature-free. Rejected: debug-only compilation (then the measured build isn't the shipped build); logging timestamps by hand to a file (the unified logging system + Instruments already visualize signpost intervals; bespoke plumbing is drift risk).
- **D5. Timer drift is verified two ways: a measured device play-through and a pinned real-clock test.** The deterministic core runner is already fully tested against a fake clock; drift can only originate in the production clock adapter (`PlayerSessionModel`'s `ContinuousClock`-driven loop) and platform scheduling. (a) **Device measurement:** play the PRD §5.4 sample routine end-to-end on the physical iPad with D4 instrumentation; from the recorded segment intervals, record in the acceptance record the maximum per-segment deviation from the compiled duration and the end-to-end deviation of total elapsed time from the compiled total. Targets: ≤ 150 ms per segment, and no cumulative growth (the end-to-end deviation must stay within the same single-segment tolerance — the PRD's "recompute from the timeline and a monotonic reference" architecture makes lateness self-correcting, and this measurement is exactly the proof). (b) **Automated integration test** in `StepBackTests`, same opt-in seam as D3: drive the production adapter with the real clock over a short compiled timeline (a few seconds total) and assert every transition lands within the 150 ms tolerance. The tolerance is never widened to make the test pass — a flaky-on-loaded-machine result is re-run on a quiet machine and, if still failing, is a real defect. Rejected: measuring on the simulator only (simulator timer coalescing is not the shipping platform); asserting drift in the fake-clock tests (they prove logic, not wall-clock behavior — the gap this milestone exists to close).

**Localization & strings**

- **D6. The string-catalog audit becomes an automated, always-on check plus the established manual renders.** Three layers: (a) **a source scan test in `StepBackTests`** that walks the app target's feature sources and fails on user-facing string literals constructed outside the catalog access layer (`StepBack/Features/Shared/L10n.swift`) — i.e. display text passed to UI or speech that does not resolve through the catalog. Non-UI literals (accessibility identifiers, `UserDefaults` keys, catalog IDs, asset names, log messages) are structurally excluded or allowlisted with a comment discipline the test documents; the allowlist is part of the reviewed diff, so a new hardcoded string cannot land silently. It also fails on hand-assembled duration/date text (the PRD §3.1 formatter rule) by flagging user-facing string interpolation of raw second/count values in feature code. (b) **Catalog compile check**: the `xcstringstool compile` dry-run already proven in the M6 gate, kept as part of the lane. (c) **Pseudolocalization walk**: double-length pseudolanguage launches across every screen — including welcome, builder with picker, stage work/rest/completion, and Settings — recorded in the acceptance record with the M3/M6 render checks as precedent (session journals 2026-07-10-1205, -1932). Acceptance #13's wording ("no user-facing strings outside the String Catalog… renders correctly under pseudolocalization") is thereby verified by construction plus render, not by grep-and-hope. Rejected: a standalone lint script outside the test lanes (checks that aren't in a gate rot); attempting full static proof of localization correctness (the scan catches the defect class PRD names; the render walk catches layout consequences).

**The manual walks**

- **D7. Two-device sync verification (#10) runs on the owner's physical iPhone + iPad, same iCloud account, via the Makefile install targets.** Procedure, each step recorded in the acceptance record: fresh install on the second device seeds nothing extra (starter dedupe by `seedIdentifier` — PRD §5.1); a routine created on device A appears on B; an edit on B appears on A; a custom workout created on A appears in B's gallery and builder picker; a session completed on B updates A's card footnote, motivation strip, and streak; deleting the custom workout on B leaves A's routine playable via the name snapshot (acceptance #8 cross-device corollary); the Settings sync line shows its M6 D11 states truthfully on both. Physical devices, not simulators: CloudKit push delivery to simulators is unreliable, and SwiftData exposes no public per-sync-event surface to script against (established in the M6 spec D11 grounding, docset `/documentation/swiftdata/syncing-model-data-across-devices`) — so sync verification is observational, on real hardware, with generous propagation patience. If a device is unpaired/locked/unreachable, the criterion is recorded BLOCKED and the gate stays honestly open (the M6-gate precedent; CLAUDE.md device rules: report, never retry blind).
- **D8. Mac end-to-end (#12): one automated walk plus the owner-visible pass.** `StepBackMacUITests` grows the full journey today's six tests only sample: sidebar browse → open routine detail → build a small routine via the builder → play in the stage window → keyboard controls (space pause/resume, arrows skip/back, escape end) → completion → the session's stats visible on the home. Resizability is asserted by exercising the stage window at two window sizes during play. The manual half — "a proper resizable stage, not a stretched phone screen" is partly a judgment call — is the owner running the installed Mac build through the same journey, recorded in the acceptance record. Rejected: manual-only (#12 regressions must fail a lane forever after, not wait for a human).
- **D9. VoiceOver end-to-end (#11) is a manual walk against a written checklist derived from the shipped specs' grouping contracts.** Build → play → complete with VoiceOver on (iPad primary, iPhone spot-check): routine card announces as one grouped element with its stats; builder step rows, rest rows, and picker tray are operable; every stage announcement mirrors the audio cues (M5's accessibility-notification contract); completion stat pairs read as grouped elements ("Streak, 5 days"); Settings rows read as label + value. Dark mode and AX-XXL Dynamic Type passes ride the same walk (acceptance #11's second clause), with the stage's documented numeral exception behaving per DESIGN.md (`stage_numerals.accessibility`: stacks vertically, countdown yields height before labels truncate). Findings are defects fixed on this branch (D10) — the checklist and outcomes land in the acceptance record. Rationale: D2's audits verify element *properties*; only a human can verify *task completability with the screen reader*, which is what #11 demands.

**Process**

- **D10. Defect vs. design-gap discipline during the walk.** Anything the walk finds where the app disagrees with an existing spec, the PRD, DESIGN.md, or ui-spec.html is a defect: fixed on this branch from the repro, under the CLAUDE.md defect rule, and the criterion re-run. Anything where the *specified* behavior itself is wrong or unspecified is a design gap: the criterion is recorded BLOCKED with the question, and the planning agent writes the amendment — the implementer never resolves a specification hole inside a verification milestone. This is the standing CLAUDE.md rule restated here because an acceptance walk is precisely where the temptation to "just make it pass" is strongest.
- **D11. `v1-candidate` is an annotated git tag on the closing commit, created only when all 13 rows read PASS.** The closing commit contains the completed acceptance record, flips this spec to Implemented, and closes the PLAN.md Milestone 7 status line; the tag message cites the acceptance record. A walk ending with any FAIL/BLOCKED rows commits the honest partial record and leaves the milestone open — PLAN.md's checkpoint rule already forbids pretending otherwise. Rejected: tagging a "mostly done" build (the tag's entire meaning is "the 13 criteria held on this exact tree").

## 5. Edge cases

Audits & tests:
- The dynamic-type audit flags the stage countdown: expected — the one D2 filter. It flags anything else on the stage (workout name, set indicator, next-up are Dynamic Type by design): real defect, fix it.
- Contrast audit on the stage: stage text is specified at ≥ 7:1 on StageCanvas (DESIGN.md `across_the_room.minimums`) — a contrast finding there is a token regression, not an exemption candidate.
- Audit findings on system-provided chrome (tab bar, toolbar items) that feature code cannot influence: investigate first; only a finding demonstrably produced by system components with no app-side fix may be filtered, and doing so amends this spec (D2's growth rule).
- Perf suite run on a loaded machine / simulator: numbers recorded as indicative only; acceptance verdicts come from device runs (D3). A device-run miss of a §8 target is a defect with a measurement attached — fix, re-measure, record both numbers.
- Drift test flake: re-run on a quiet machine; persistent > 150 ms transitions are a real player defect (D5's no-widening rule).
- The welcome-audit lane and existing UI-test seams interact: audits run with the standard `-StepBackUITesting` bootstrap (fake sync status, in-memory store) so no lane touches CloudKit or real defaults.

Manual walks & devices:
- iPad or iPhone unavailable for D7: record BLOCKED, gate stays open, ask the owner to connect/unlock — never substitute simulators for the sync verdict.
- iCloud degraded mid-verification (no account / temporarily unavailable): the Settings line showing its degraded state *is* part of the #10 evidence; the propagation steps wait for restored availability rather than recording a false FAIL.
- Sync propagation is slow but eventual: allow generous time and app-foreground cycles before judging a step failed (CloudKit mirroring batches; there is no progress surface by design).
- A criterion is unmeetable as written (e.g. an acceptance step contradicts shipped, specced behavior): design gap → D10 planner path; the record says BLOCKED with the exact conflict.
- Pseudolocalization reveals clipping/truncation: layout defect on this branch; re-render after the fix and record the second pass.
- The two uncommitted Mac fixes on the M6 branch (header note): if the owner drops rather than commits them, the Mac walk (D8) will re-find any real problems they fixed — the walk must run against what is actually committed, never against loose work.

Instrumentation:
- Signposts must never alter timing they measure: interval calls sit outside the segment-advance math, and the drift measurement (D5) is itself the regression check that instrumentation stayed free.
- Signpost data absent in a release-configuration spot-check: verify with Instruments that intervals appear on the Debug device build used for measurement; the acceptance record names the configuration measured.

## 6. Accessibility & localization

- **New string-catalog keys: none.** This milestone adds no user-facing text — an empty key table is the point. Any hardcoded string the D6 scan or pseudolocalization walk uncovers is a defect: the fix moves it into `Localizable.xcstrings` under the normal key-naming conventions, and the acceptance record lists every key added this way (they are findings, not features).
- **New accessibility identifiers: none planned.** If an audit or the Mac walk needs a stable handle that doesn't exist, it follows the established `screen.element` naming and is listed in the acceptance record.
- **VoiceOver:** no new groupings — D9 verifies the groupings the M3–M6 specs already contracted. The D2 audits add no VoiceOver-visible surface.
- **Dynamic Type:** no new exceptions; D2/D9 verify the existing single exception behaves per its documented AX contract.
