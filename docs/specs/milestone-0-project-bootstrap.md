## 1. Problem

The repo is a docs-only scaffold (PRD, DESIGN, ui-spec, PLAN). Nothing can be built, tested, or run. Every later milestone assumes: a generated Xcode project with app + test targets for iPad/iPhone/Mac, a local pure-logic Swift package with a standalone test lane, the design-token asset catalog, an empty String Catalog, and repeatable `make` commands. PLAN.md Milestone 0 defines the gate: simulator build succeeds on iPad + iPhone destinations and macOS, and empty test suites pass.

## 2. Goals

- G1. `project.yml` + xcodegen generation of `StepBack.xcodeproj`, mirroring the proven intelli-expense shape (per CLAUDE.md), adapted to this product: iPad-first universal iOS target plus a native macOS target, both on OS 26.0+, Swift 6 strict concurrency.
- G2. Local Swift package **StepBackCore** (pure logic home for the timeline compiler, stats math, catalog decoding in Milestone 1) with its own test target; `swift test` runs standalone from the package directory.
- G3. Asset catalog carrying every DESIGN.md color token as named color sets, plus `AccentColor` aliasing PulseAzure; empty `Localizable.xcstrings`.
- G4. Unit + UI test targets that compile and pass while empty (one placeholder test each, so "the suite runs green" is demonstrable rather than vacuous).
- G5. `Makefile` with the repeatable commands; CLAUDE.md §Commands updated to match.
- G6. Verified toolchain assumptions recorded (xcodegen ≥ 2.45, Xcode 26.5, OS 26.x simulators — all confirmed present on the build machine at spec time).

## 3. Non-goals

- No feature UI, no navigation scaffold, no tab bar — the app target's root is a deliberately blank placeholder (D7). Screens start at Milestone 3.
- No SwiftData models, no CloudKit entitlement or container, no iCloud capability (Milestone 2). Deferring the CloudKit entitlement keeps M0 signing trivial; the day-1 CloudKit *schema* rules in CLAUDE.md constrain model design, and no models exist yet.
- No catalog JSON, no compiler/runner/stats code in StepBackCore beyond a compilable placeholder (Milestone 1 owns those, TDD).
- No app icon design (a placeholder/default icon slot only), no launch-screen artwork, no App Store/TestFlight/provisioning lanes (intelli-expense's `asc`/archive targets are out of scope until a distribution milestone exists).
- No third app target (no sandbox variant, no extensions) — PRD §2 non-goals exclude widgets/extensions in v1; the sandbox-lane pattern from intelli-expense is not needed until there is seeded data worth screenshotting.
- Checked against PRD §2: nothing here adds programs, metrics, media content, or network surface; this is pure scaffolding.

## 4. Design decisions

- **D1. Target shape: two app targets sharing one source folder — `StepBack` (iOS, `TARGETED_DEVICE_FAMILY` "1,2") and `StepBackMac` (native macOS) — not a single xcodegen multiplatform target.** PLAN.md says "one multiplatform app target … mirroring the intelli-expense shape"; those two clauses conflict, and the intelli-expense shape wins because CLAUDE.md names it the proven reference: intelli-expense ships separate iOS and macOS targets consuming the same source tree with per-platform Info.plists and (later) entitlements, and its Makefile/scheme lanes assume that split. Rejected alternative: xcodegen `supportedDestinations`-style single target — less proven in the house, and per-platform divergence (Mac keyboard-stage scene, Mac Info.plist keys, future compilation conditions like intelli-expense's `INTELLI_EXPENSE_MAC`) is cleaner with explicit targets. The key difference from intelli-expense: device family is "1,2" (iPad-first product; intelli-expense is iPhone-only "1"). PRD §3 requires size-class-driven layout with no device-type branches, which is orthogonal to target count.
- **D2. Project generation and hygiene mirror intelli-expense:** `xcodegen generate` from `project.yml` (minimum xcodegen version pinned ≥ 2.43 as intelli-expense does; 2.45.3 is installed), the generated `.xcodeproj` **committed** (intelli-expense tracks it; only `xcuserdata`, `build/`, DerivedData, `.build/`, `*.xcresult` are gitignored), `make gen` regenerates. Deployment target 26.0 for iOS and macOS; `SWIFT_VERSION` 6.0 with `SWIFT_STRICT_CONCURRENCY: complete` at the settings base, exactly as the reference project.
- **D3. StepBackCore package:** `swift-tools-version: 6.2` (matches intelli-expense's ExpenseCore; installed toolchain is Swift 6.3.2), platforms iOS 26 / macOS 26, one library product, one target plus one test target with a `Fixtures` resource folder wired via `resources: .process` from day 1 (Milestone 1's trimmed-catalog fixtures land there without touching the manifest). The package contains a single compilable placeholder type and no logic — Milestone 1 is TDD and must start from a green empty package. Both app targets depend on the package.
- **D4. Asset catalog tokens — the full DESIGN.md inventory, verbatim hex, one named color set per token.** Twelve light/dark pairs (Any + Dark appearances per color set, the intelli-expense `AccentColor.colorset` JSON shape): `PulseAzure` (#0B63E5 / #3D9BFF), `PulseAzureSoft` (#E8F0FD / rgba(61,155,255,0.16)), `RecoverMint` (#0E9F8E / #3BD6C3), `RecoverMintSoft` (#E6F4F1 / rgba(59,214,195,0.14)), and the eight category hues `CategoryFullBody` (#5856D6 / #5E5CE6), `CategoryCore` (#FF9500 / #FF9F0A), `CategoryArmsShoulders` (#AF52DE / #BF5AF2), `CategoryChestBack` (#1E96D1 / #4CC2FF), `CategoryLegsGlutes` (#96419B / #D678DB), `CategoryCardio` (#B26B00 / #E5A421), `CategoryMobilityStretch` (#5D6B85 / #93A2C0), `CategoryBalance` (#A2845E / #AC8E68). Four appearance-invariant stage sets (single Any value, **no** Dark variant — DESIGN.md: "Stage tokens are appearance-invariant single values … not a forced-dark override hack"): `StageCanvas` (#0B0F14), `StageSurface` (#161C24), `StageText` (#F5F7FA), `StageTextDim` (#F5F7FA at 80% alpha, strengthened by the Milestone 7 accessibility remediation). Plus `AccentColor` whose Any/Dark components byte-match PulseAzure — DESIGN.md: "`AccentColor` in the asset catalog byte-matches `PulseAzure`; it is an alias, not a second token"; wired via `ASSETCATALOG_COMPILER_GLOBAL_ACCENT_COLOR_NAME` as in the reference project. This satisfies the DESIGN.md rules "Every custom token is a light/dark PAIR — one asset-catalog color set per token. Code references the named color only" and "no raw hex and no colorScheme branches in feature code" — the catalog is what makes those rules satisfiable from the first UI commit. Naming: UpperCamelCase of the DESIGN.md token names; category sets named after the category (not the hue), because DESIGN.md binds hue to category identity permanently.
- **D5. String Catalog:** `Localizable.xcstrings` created as the minimal valid empty catalog (top-level `sourceLanguage` "en", empty `strings`, `version` "1.0" — the exact top-level shape of intelli-expense's catalog), added as a resource to both app targets. It stays genuinely empty this milestone because M0 ships no user-facing text (D7); PRD §3.1's "every user-facing string from the first commit" is honored by having the catalog exist before the first string does.
- **D6. Test frameworks: Swift Testing for all unit tests; XCUIAutomation for UI tests.** Verified against the offline Apple docset: Swift Testing (`/documentation/testing`, Platforms: Swift 6.0+, Xcode 16.0+) integrates with both SPM (`swift test`) and Xcode test bundles; UI automation (`XCUIApplication`) lives in the XCUIAutomation framework (`/documentation/xcuiautomation`, split from XCTest as of Xcode 16.3) and has no Swift Testing equivalent, so `StepBackUITests`/`StepBackMacUITests` remain XCTest-style UI test bundles. This deliberately diverges from intelli-expense (whose unit tests predate the house adopting Swift Testing): a greenfield 2026 project takes Apple's current default for new unit tests; the TDD core in Milestone 1 benefits from parameterized tests (timeline edge-case tables). Placeholder content per target: one trivially-true Swift Testing test in StepBackCoreTests and StepBackTests; one launch smoke test in each UI bundle (app launches, foreground state reached — no UI assertions, since the root is a blank placeholder).
- **D7. App target content: an app entry point with an intentionally empty root view; no user-facing strings, no accessibility identifiers, nothing rendered.** Rationale: any visible text would either bypass the String Catalog (a PRD §3.1 defect) or force premature UI decisions that belong to Milestone 3's spec (tab scaffold is explicitly a Milestone 3 deliverable). Rejected alternative: a "Hello StepBack" splash — it would be the app's first string, first screen, and first drift risk, for zero gate value. The design-system rules therefore constrain nothing yet; no DESIGN.md quote is applicable to a deliberately blank screen, and that is the point.
- **D8. Signing: automatic, using the house development team supplied through local configuration, bundle IDs `com.nags.stepback`, `com.nags.stepback.tests`, `com.nags.stepback.uitests`, `com.nags.stepback.macuitests`; Mac target reuses `com.nags.stepback`** (intelli-expense pattern: iOS and Mac apps share the bundle ID as one product). Manual signing/profiles are an App Store concern deferred with distribution. No entitlements files in M0 (no CloudKit yet, D-non-goal); the Mac target gets the default app sandbox entitlement only if the 26.5 toolchain requires one to build — if so it is committed and noted, otherwise no entitlements exist yet.
- **D9. Makefile lanes (and the CLAUDE.md §Commands text) — the intelli-expense verbs, trimmed to what exists:** `gen` (xcodegen), `test-core` (`swift test` in StepBackCore), `test-app` (iOS scheme tests on the iPhone simulator), `test-ipad` (same tests on the iPad simulator — StepBack is iPad-first, so the iPad destination is a first-class lane, not an afterthought), `test-mac`, `build-sim` (iPhone), `build-sim-ipad`, `build-mac`, `test` (aggregate: core + app). Default simulator destinations pinned to devices present on the build machine: iPhone 17 Pro and iPad Pro 11-inch (M5), OS 26.x, overridable via variables as in the reference Makefile. Schemes: `StepBack` (builds app, tests unit + UI) and `StepBackMac` (builds Mac app, tests Mac UI bundle).
- **D10. `.claude/launch.json` keeps the existing `design-spec` server entry unchanged;** no app-run configuration is added until there is something to preview (preview tooling can't drive a simulator anyway).

## 5. Edge cases

- **Empty-suite gate honesty:** a test target with zero tests can pass vacuously or be silently skipped by scheme config. The placeholder tests (D6) exist precisely so the gate proves the test *pipelines* work end-to-end (compile, host-app injection for unit tests, launch for UI tests).
- **xcodegen regeneration drift:** the committed `.xcodeproj` and `project.yml` can diverge if someone edits the project in Xcode. Rule (inherited from the reference project, restate in CLAUDE.md §Commands): `project.yml` is the source of truth; run `make gen` after changing it; never hand-edit the project file.
- **Stage tokens must not gain Dark variants** by reflex when created in Xcode's editor — a Dark appearance on a stage color would silently reintroduce appearance-dependence that DESIGN.md forbids. The four stage sets carry a single universal value.
- **Soft-token dark values are rgba with alpha** (PulseAzureSoft, RecoverMintSoft, StageTextDim); the color-set components must carry the alpha, not a flattened opaque approximation.
- **Simulator availability:** destinations are pinned by name, not UDID; if the named device is missing on another machine the Make variable overrides (`SIM_DEST`-style) are the escape hatch — document them in the Makefile help text.
- **`swift test` vs Xcode:** StepBackCore must test green both standalone (`make test-core`) and when its tests are not part of the app scheme — the package's tests are not added to the app scheme (they run in their own lane), mirroring the reference project.
- **First-build package resolution:** StepBackCore is path-local; no network access is required for any build or test lane (consistent with the product's zero-network posture and with CI-less local development).

## 6. Accessibility & localization

- No user-facing strings, no visible UI, no VoiceOver surface ships in this milestone (D7). New accessibility identifiers: none. New string-catalog keys: none — the deliverable is the empty `Localizable.xcstrings` itself, present and wired into both app targets so Milestone 3's strings have a home from their first commit.
- The asset-catalog tokens are the accessibility groundwork: DESIGN.md's contrast rules (e.g. stage text ≥ 7:1 on StageCanvas) become verifiable in code only because the tokens exist as named sets.

## 7. Test impact

- New: `StepBackCoreTests` (Swift Testing, one placeholder test, `swift test` standalone), `StepBackTests` (Swift Testing, one placeholder test, runs in the `StepBack` scheme), `StepBackUITests` + `StepBackMacUITests` (XCUIAutomation launch smoke tests).
- No existing tests to migrate (there are none).
- The verification gate for this spec is the PLAN.md Milestone 0 gate: `make gen && make test-core && make build-sim && make build-sim-ipad && make build-mac && make test-app && make test-mac` all green.

## 8. Acceptance criteria

1. `make gen` regenerates `StepBack.xcodeproj` from `project.yml` deterministically; the generated project is committed.
2. `make build-sim` (iPhone 17 Pro class destination), `make build-sim-ipad` (iPad Pro 11-inch class destination), and `make build-mac` all succeed from a clean checkout.
3. `make test-core` runs the StepBackCore package tests standalone via `swift test` and passes; `make test-app` and `make test-mac` pass, executing the placeholder unit and UI tests (test count ≥ 1 in every bundle — no vacuously empty suite).
4. The asset catalog contains exactly the D4 inventory: 12 light/dark pair color sets + 4 single-value stage sets + `AccentColor` byte-matching PulseAzure in both appearances; hex values match DESIGN.md verbatim; stage sets have no Dark appearance.
5. `StepBack/Resources/Localizable.xcstrings` exists, is valid (Xcode opens it; source language `en`), contains zero strings, and is a resource of both app targets.
6. Both app targets compile with Swift 6 strict concurrency (`complete`); deployment targets are 26.0 across iOS and macOS; the iOS target's device family is iPhone + iPad.
7. The app launches to a blank root on iPad simulator, iPhone simulator, and Mac without crash (demonstrated by the UI smoke tests); no user-facing string literals exist anywhere in app sources.
8. CLAUDE.md §Commands lists the real Make lanes (replacing the "will be added" placeholder), including the regeneration rule from Edge cases; the CLAUDE.md token-count correction (Docs-amends header) is applied.
9. All work lands as one coherent commit on `codex/milestone-0-project-bootstrap` (plus the PLAN.md/spec status flips per the workflow rules); the tree is clean afterward.
